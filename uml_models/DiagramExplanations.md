# Alert Generation System

The Alert Generation System monitors incoming patient vital signs and triggers alerts when abnormal values are detected.

The AlertGenerator evaluates incoming PatientData and checks it regarding the thresholds stored in the ThresholdManager. Each patient has different threshold rules stored in the ThresholdRule which is managed by ThresholdManager. A ThresholdManager contains at least one ThresholdRule.

When a threshold is violated, an Alert is generated by the AlertGenerator containing the details: alertId, patiendId, condition and timestamp. Then AlertManager then forwards the created alert to the medical staff.

This system ensures that whenever a patient data becomes abnormal, the medical staff recevies the regarding alert, ensuring a fast intervention.

# Data Storage System

The Data Storage System is responsible for storing and managing all incoming patient data.

DataStorage recevies and stores PatientData records which include vital signs, timestamps, and version information. A DataStorage can contain multiple PatientData entries. Them DataRetriever is used to query stored data by patientId or by time range.

Access to the data is controlled by AccessController which checks the user roles before granting access to ensure that the patient data privacy. Every access or modification is recorded in the AuditLog to ensure traceability.

Old data is automatically removed by RetentionPolicy after a defined number of days to manage storage size and ensure data privacy.

This system ensures that patient data is securely stored, easily retrievable, and properly controlled.

# Patient Identification System

The Patient Identification System ensures that incoming data is correctly matched to the right patient.

PatientIdentifier uses the patientId from incoming data to find the corresponding HospitalPatient in the HospitalDatabase. If the match is validated, a HospitalPatient object is returned containing the patient details.

IdentityManager checks the match and handles incorrect or missing patient cases, storing them as errors. If a patient cannot be found or an error occurs, an IdentityError record is created and stored for tracking and debugging.

This system ensures that all patient data is correctly linked before being processed by other parts of the system.

# Data Access Layer

The Data Access Layer handles incoming data from external sources and prepares it for processing inside the system.

DataListener is an abstract implemented by TCPDataListener, WebSocketDataListener, and FileDataListener to support multiple input sources. Each listener receives raw data from its respective source.

The raw data is then passed to DataParser, which parses it into a standard PatientData format.

DataSourceAdapter then hands the parsed data to DataStorage.

This system ensures that different data sources can be added and parsed throughly before getting forwarded to the data storage.
